## 🔧 Technical Implementation

### 1. GitHub Actions Workflow

**File:** `.github/workflows/ci.yml`

**Key Features:**
- Sequential job execution (lint → build → deploy)
- Caching for `node_modules` (faster builds)
- Artifact upload (build output retention)
- Automatic PR comments on conflicts

### 2. Lint & Type Check Job

```yaml
lint:
  runs-on: ubuntu-latest
  steps:
    - uses: actions/checkout@v4
    - uses: actions/setup-node@v4
      with:
        node-version: '20'
        cache: 'npm'
    - run: npm ci
    - run: npm run lint        # ESLint
    - run: npx tsc --noEmit   # TypeScript
```

**Testing Results:**
- ✅ ESLint: 1 warning (non-blocking)
- ✅ TypeScript: No errors
- **CI Status**: Will pass

### 3. Build Verification Job

```yaml
build:
  needs: lint
  runs-on: ubuntu-latest
  steps:
    - uses: actions/checkout@v4
    - uses: actions/setup-node@v4
      with:
        node-version: '20'
        cache: 'npm'
    - run: npm ci
    - run: npm run build
    - uses: actions/upload-artifact@v4
      with:
        name: build-output
        path: dist/
        retention-days: 7
```

**Build Results:**
- ✅ Bundle size: 318.60 kB (gzip: 103.54 kB)
- ✅ CSS size: 4.49 kB (gzip: 1.34 kB)
- ✅ Build time: 515ms
- **CI Status**: Will pass

### 4. Merge Conflict Detection

```yaml
conflict-check:
  runs-on: ubuntu-latest
  steps:
    - uses: actions/checkout@v4
      with:
        fetch-depth: 0
    - run: |
        git fetch origin main
        git merge origin/main --no-commit --no-ff
        if [ $? -ne 0 ]; then
          echo "❌ Merge conflicts detected!"
          exit 1
        fi
    - uses: actions/github-script@v7
      if: failure()
      with:
        script: |
          github.rest.issues.createComment({
            body: '⚠️ **Merge conflicts detected** with `main`'
          })
```

**Behavior:**
- Automatically detects conflicts before merge
- Comments on PR with conflict details
- Blocks merge until resolved

## 🎯 Key Technical Decisions

### Why Sequential Jobs?

```yaml
lint → build → deploy
```

**Benefits:**
- Fail fast (linting catches issues before build)
- Clear dependency chain
- Faster feedback (no waiting for unnecessary steps)

### Why npm ci over npm install?

```yaml
- run: npm ci
```

**Benefits:**
- Consistent installs (uses package-lock.json)
- Faster (skips dependency resolution)
- CI best practice

### Why Artifact Upload?

```yaml
- uses: actions/upload-artifact@v4
  with:
    name: build-output
    path: dist/
    retention-days: 7
```

**Benefits:**
- Debug failed deployments
- Compare build sizes
- Historical tracking

## 🎓 Lessons Learned

### CI/CD Best Practices

1. **Cache node_modules**
   ```yaml
   - uses: actions/setup-node@v4
     with:
       cache: 'npm'
   ```
   - Faster installs
   - Reduced bandwidth
   - Consistent builds

2. **Use npm ci in CI**
   - Reproducible builds
   - Faster than npm install
   - Uses lockfile

3. **Fail Fast**
   - Lint before build
   - Type check before deploy
   - Clear errors early

4. **Comment on PR**
   - Automatic feedback
   - No need to check CI tab
   - Better UX for developers

### Frontend Tooling

1. **ESLint > TypeScript for linting**
   - Faster feedback
   - Configurable rules
   - Easier to fix

2. **TypeScript for type safety**
   - Catch errors at compile time
   - Better IDE support
   - Safer refactoring

3. **Vite for building**
   - Fast HMR in dev
   - Optimized production builds
   - Modern tooling
